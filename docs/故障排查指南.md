# GoPay 故障排查指南

## 1. 服务无法启动

### 症状：启动时报 "Failed to connect to database"

排查步骤：
1. 检查数据库是否运行：`podman compose --env-file .env ps postgres`
2. 检查连接参数：`DB_HOST`、`DB_PORT`、`DB_USER`、`DB_PASSWORD`
3. 测试连接：`psql -h $DB_HOST -U $DB_USER -d $DB_NAME`
4. 检查防火墙/安全组是否放行 5432 端口

### 症状：启动时报 "Failed to load config"

排查步骤：
1. 检查 `.env` 文件是否存在
2. 检查必填环境变量是否设置：`DB_HOST`、`DB_USER`、`DB_PASSWORD`、`DB_NAME`、`MASTER_KEY`
3. 检查 `MASTER_KEY` 是否为有效的 Base64 字符串

### 症状：启动时报 "Failed to run migrations"

排查步骤：
1. 检查数据库用户是否有 DDL 权限
2. 检查 `migrations/` 目录是否存在
3. 手动执行迁移脚本查看具体错误

## 2. 支付失败

### 症状：创建订单返回 "channel not found"

原因：支付渠道未配置或未激活。

解决：
```sql
-- 检查渠道配置
SELECT app_id, channel, status FROM channel_configs WHERE app_id = 'your_app_id';

-- 激活渠道
UPDATE channel_configs SET status = 'active' WHERE app_id = 'your_app_id' AND channel = 'wechat_native';
```

### 症状：创建订单返回 "app not found"

原因：应用未注册。

解决：
```sql
-- 检查应用
SELECT app_id, status FROM apps WHERE app_id = 'your_app_id';

-- 确认应用状态为 active
```

### 症状：微信支付返回签名错误

排查步骤：
1. 检查商户私钥文件是否正确：`WECHAT_PRIVATE_KEY_PATH`
2. 检查证书序列号是否匹配：`WECHAT_SERIAL_NO`
3. 检查 API v3 密钥是否正确：`WECHAT_API_V3_KEY`
4. 确认证书未过期：`openssl x509 -in certs/wechat/apiclient_cert.pem -noout -dates`

### 症状：支付宝返回签名验证失败

排查步骤：
1. 检查应用私钥是否正确：`ALIPAY_APP_PRIVATE_KEY`
2. 检查支付宝公钥是否正确：`ALIPAY_PUBLIC_KEY`
3. 确认使用的是 RSA2 签名方式
4. 检查是否使用了正确的网关地址（沙箱 vs 生产）

## 3. 回调通知失败

### 症状：订单已支付但业务系统未收到通知

排查步骤：
1. 检查订单通知状态：
```sql
SELECT order_no, notify_status, retry_count FROM orders WHERE order_no = 'xxx';
```

2. 检查通知日志：
```sql
SELECT * FROM notify_logs WHERE order_no = 'xxx' ORDER BY created_at DESC;
```

3. 检查业务系统回调 URL 是否可达
4. 手动重试：
```bash
curl -X POST http://localhost:8080/internal/api/v1/orders/{order_no}/retry \
  -H "X-API-Key: your-admin-api-key"
```

### 症状：通知一直重试失败

常见原因：
- 业务系统回调 URL 不可达
- 业务系统返回非 200 状态码
- 业务系统响应超时（> 3 秒）
- 回调 URL 配置错误

解决：
1. 确认回调 URL 可从 GoPay 服务器访问
2. 确认业务系统在 3 秒内返回 HTTP 200
3. 检查业务系统日志

## 4. 对账异常

### 症状：对账报告显示"长款"（平台有、本地无）

可能原因：
- Webhook 回调未成功处理
- 订单状态更新失败

解决：
1. 检查对应时间段的 Webhook 日志
2. 手动查询平台订单状态
3. 手动更新本地订单状态

### 症状：对账报告显示"短款"（本地有、平台无）

可能原因：
- 订单已创建但用户未支付
- 平台账单延迟

解决：
1. 检查订单状态是否为 pending
2. 等待 T+2 再次对账确认

### 症状：对账报告显示"金额不匹配"

可能原因：
- 部分退款导致金额差异
- 数据记录错误

解决：
1. 检查是否有退款记录
2. 对比平台后台的实际金额

## 5. 性能问题

### 症状：响应时间变长

排查步骤：
1. 检查数据库连接池：
```sql
SELECT count(*) FROM pg_stat_activity WHERE datname = 'gopay';
```

2. 检查慢查询：
```sql
SELECT query, mean_exec_time, calls
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;
```

3. 检查表大小和索引：
```sql
SELECT relname, n_live_tup, n_dead_tup
FROM pg_stat_user_tables
ORDER BY n_live_tup DESC;
```

4. 检查 Prometheus 指标中的请求延迟分布

### 症状：数据库连接耗尽

解决：
1. 增加连接池大小：`DB_MAX_OPEN_CONNS=100`
2. 检查是否有连接泄漏（长时间 idle 连接）
3. 检查是否有长事务阻塞

## 6. 安全问题

### 症状：收到证书过期告警

处理：
1. 从支付平台下载新证书
2. 替换证书文件
3. 重启服务
4. 验证新证书生效

### 症状：API Key 泄露

处理：
1. 立即更换 `ADMIN_API_KEY`
2. 重启服务
3. 检查审计日志，确认是否有异常操作
4. 如有必要，轮换 `MASTER_KEY`

### 症状：限流触发

说明 IP 请求频率超过阈值（默认 100 req/s）。

排查：
1. 检查是否为正常业务流量
2. 如果是攻击，考虑在网关层封禁 IP
3. 如果是正常流量，调整限流参数

## 7. 数据一致性问题

### 症状：订单状态不一致

**场景 1：支付平台显示已支付，GoPay 显示 pending**

排查步骤：
```sql
-- 1. 查询订单状态
SELECT order_no, status, channel_order_no, paid_at, updated_at
FROM orders
WHERE out_trade_no = 'BIZ_ORDER_001';

-- 2. 检查 Webhook 日志
SELECT * FROM notify_logs
WHERE order_no = 'ORD_xxx'
ORDER BY created_at DESC;

-- 3. 检查交易流水
SELECT * FROM transactions
WHERE order_no = 'ORD_xxx'
ORDER BY created_at DESC;
```

解决方案：
```bash
# 手动查询支付平台订单状态
curl -X GET http://localhost:8080/internal/api/v1/orders/{order_no}/sync \
  -H "X-API-Key: your-admin-api-key"

# 或手动更新订单状态（谨慎操作）
UPDATE orders
SET status = 'paid', paid_at = NOW()
WHERE order_no = 'ORD_xxx' AND status = 'pending';
```

**场景 2：重复支付**

排查步骤：
```sql
-- 查找相同 out_trade_no 的订单
SELECT order_no, out_trade_no, status, amount, created_at
FROM orders
WHERE out_trade_no = 'BIZ_ORDER_001'
ORDER BY created_at;

-- 检查是否有多次支付记录
SELECT COUNT(*), out_trade_no
FROM orders
WHERE status = 'paid'
GROUP BY out_trade_no
HAVING COUNT(*) > 1;
```

解决方案：
1. 确认哪笔是有效支付
2. 对多余的支付发起退款
3. 检查业务系统是否有重复下单逻辑

### 症状：金额不匹配

排查步骤：
```sql
-- 查询订单金额
SELECT order_no, out_trade_no, amount, channel_order_no
FROM orders
WHERE order_no = 'ORD_xxx';

-- 对比交易流水
SELECT transaction_no, amount, type, status
FROM transactions
WHERE order_no = 'ORD_xxx';
```

常见原因：
- 业务系统传入金额单位错误（元 vs 分）
- 退款后金额未更新
- 数据库字段类型溢出

## 8. 并发问题

### 症状：订单重复创建

原因：并发请求导致唯一索引失效前创建了多个订单。

排查：
```sql
-- 查找重复订单
SELECT app_id, out_trade_no, COUNT(*)
FROM orders
GROUP BY app_id, out_trade_no
HAVING COUNT(*) > 1;
```

解决：
1. 确认数据库唯一索引存在：
```sql
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'orders' AND indexdef LIKE '%out_trade_no%';
```

2. 如果索引缺失，创建索引：
```sql
CREATE UNIQUE INDEX idx_orders_app_out_trade
ON orders(app_id, out_trade_no);
```

### 症状：死锁

日志显示：`deadlock detected`

排查：
```sql
-- 查看当前锁等待
SELECT
  blocked_locks.pid AS blocked_pid,
  blocked_activity.usename AS blocked_user,
  blocking_locks.pid AS blocking_pid,
  blocking_activity.usename AS blocking_user,
  blocked_activity.query AS blocked_statement,
  blocking_activity.query AS blocking_statement
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_activity.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks ON blocking_locks.locktype = blocked_locks.locktype
JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_activity.pid = blocking_locks.pid
WHERE NOT blocked_locks.granted;

-- 查看长时间运行的事务
SELECT pid, now() - pg_stat_activity.query_start AS duration, query, state
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY duration DESC;
```

解决：
1. 优化事务逻辑，减少锁持有时间
2. 调整锁顺序，避免循环等待
3. 必要时终止阻塞事务：
```sql
SELECT pg_terminate_backend(pid);
```

## 9. 内存和资源问题

### 症状：内存持续增长（内存泄漏）

排查步骤：
```bash
# 1. 查看进程内存使用
podman stats gopay

# 2. 使用 pprof 分析内存
curl http://localhost:8080/debug/pprof/heap > heap.prof
go tool pprof -http=:8081 heap.prof

# 3. 查看 goroutine 数量
curl http://localhost:8080/debug/pprof/goroutine?debug=1
```

常见原因：
- HTTP 连接未关闭
- 数据库连接泄漏
- goroutine 泄漏
- 缓存未设置过期时间

解决：
1. 确保所有 HTTP Response Body 都被关闭
2. 使用 `defer resp.Body.Close()`
3. 检查 goroutine 是否正常退出
4. 设置合理的缓存过期时间

### 症状：CPU 使用率过高

排查步骤：
```bash
# 1. 查看 CPU 使用情况
top -p $(pgrep gopay)

# 2. 使用 pprof 分析 CPU
curl http://localhost:8080/debug/pprof/profile?seconds=30 > cpu.prof
go tool pprof -http=:8081 cpu.prof

# 3. 查看热点函数
go tool pprof -top cpu.prof
```

常见原因：
- 死循环
- 大量正则表达式匹配
- JSON 序列化/反序列化
- 加密解密操作

### 症状：文件描述符耗尽

日志显示：`too many open files`

排查：
```bash
# 查看当前打开的文件数
lsof -p $(pgrep gopay) | wc -l

# 查看限制
ulimit -n

# 查看具体打开的文件
lsof -p $(pgrep gopay) | head -50
```

解决：
```bash
# 临时提高限制
ulimit -n 65535

# 永久修改（编辑 /etc/security/limits.conf）
* soft nofile 65535
* hard nofile 65535

# Docker 容器中修改
# 在 docker-compose.yml 中添加
ulimits:
  nofile:
    soft: 65535
    hard: 65535
```

## 10. 网络问题

### 症状：连接超时

**场景 1：连接支付平台超时**

排查：
```bash
# 测试网络连通性
curl -v https://api.mch.weixin.qq.com
curl -v https://openapi.alipay.com

# 检查 DNS 解析
nslookup api.mch.weixin.qq.com

# 检查路由
traceroute api.mch.weixin.qq.com
```

解决：
1. 检查防火墙规则
2. 检查出站网络策略
3. 配置 HTTP 代理（如需要）
4. 增加超时时间

**场景 2：业务系统回调超时**

排查：
```bash
# 测试回调 URL 可达性
curl -v -X POST https://your-business.com/callback \
  -H "Content-Type: application/json" \
  -d '{"test": "data"}'

# 检查响应时间
time curl -X POST https://your-business.com/callback
```

解决：
1. 优化业务系统回调接口性能
2. 确保回调接口在 3 秒内响应
3. 使用异步处理，立即返回 200

## 11. 常用排查命令速查

```bash
# 服务状态
podman compose --env-file .env ps
systemctl status gopay

# 日志查看
podman compose --env-file .env logs -f gopay
tail -f /var/log/gopay/gopay.log

# 数据库连接
psql -h localhost -U gopay -d gopay

# 性能分析
curl http://localhost:8080/debug/pprof/profile?seconds=30 > cpu.prof
curl http://localhost:8080/debug/pprof/heap > heap.prof

# 网络测试
curl -v http://localhost:8080/health
telnet localhost 8080

# 资源监控
top -p $(pgrep gopay)
podman stats gopay
lsof -p $(pgrep gopay) | wc -l

# 数据库查询
# 查看活跃连接
SELECT * FROM pg_stat_activity WHERE datname = 'gopay';

# 查看慢查询
SELECT query, mean_exec_time FROM pg_stat_statements ORDER BY mean_exec_time DESC LIMIT 10;

# 查看表大小
SELECT relname, pg_size_pretty(pg_total_relation_size(relid)) FROM pg_stat_user_tables ORDER BY pg_total_relation_size(relid) DESC;
```
