# GoPay 内部统一支付网关 - 产品需求文档 (PRD)

## 一、 项目概览

- **项目名称：** GoPay
    
- **项目定位：** 极简、高性能、全语言适配的内部支付中台。
    
- **核心目标：** 
    1. **成本优化：** 只支付第三方支付平台的基础费率（0.6%），不额外支付聚合支付服务商费用（节省 0.2-0.5%）
    2. **一个商户号管理多业务：** 通过 `app_id` 机制，让一个微信/支付宝商户号服务多个独立业务线，降低商户申请成本和管理复杂度
    3. **开源工具定位：** 为像我一样的一人公司提供可自部署的支付聚合工具，未来可开源或 SaaS 化
    
- **为什么不用 Ping++/BeeCloud：**
    - 额外费率 0.2-0.5%，年交易额 100 万就多付 2000-5000 元
    - 无法优雅地实现"一个商户号对应多套业务"
    - 代码不在自己手里，无法开源或二次开发
    

## 二、 核心架构原则

1. **极致解耦：** 业务系统（上层）只负责业务逻辑，GoPay（底层）只负责支付通道联调与资金状态同步。
    
2. **协议中立：** 内部通信采用标准 JSON over HTTP，异步通知采用 HTTP 回调（带重试机制）。
    
3. **资金安全：** 不设实时复式账本，通过 **T+1 离线对账** 确保内部数据与第三方平台资金流的绝对一致。

4. **极简架构：** 只依赖 PostgreSQL 一个外部组件，砍掉 Redis 和 RabbitMQ，降低运维复杂度。
    

## 三、 业务模型 (Domain Model)

- **App (业务应用)：** 独立的产品线（如：酒店竞价系统、理财沙箱）。拥有独立的 `AppID` 和支付渠道配置。
    
- **Order (支付订单)：** GoPay 生成的统一流水。关联业务侧订单号与第三方支付平台流水号。
    
- **Channel (支付渠道)：** 如 `wechat_pay`, `alipay`, `stripe`。
    
- **Reconciliation (对账单)：** T+1 生成的核对报告，记录长款、短款及金额差异。
    

## 四、 功能需求

### 1. 统一下单链路 (Standard Checkout)

- **功能描述：** 提供标准 API 供各业务线调用。
    
- **流程：** 业务侧 POST -> GoPay 校验身份 -> 加密提取商户证书 -> 调用第三方接口 -> 记录 `Pending` 订单 -> 返回支付参数。
    

### 2. 统一回调与事件分发 (Webhook & HTTP Callback)

- **功能描述：** 接收第三方平台通知，验签后通过异步 HTTP 回调通知业务系统。
    
- **极致规范：**
    
    - **强验签：** 使用生产环境环境变量注入的 Master Key 对商户密钥进行解密后再验签。
        
    - **事务原子性：** 确保支付单状态更新在事务内完成，事务提交后再异步发送 HTTP 通知。
        
    - **铁律一：绝不允许"跨网络请求"持有数据库事务。** 必须先提交事务释放数据库连接，再异步通知业务系统。
        
    - **铁律二：异步回调必须有严酷的"超时与重试上限"。** HTTP 请求强制 3 秒超时，最多重试 5 次（指数退避），失败后标记为 `FAILED_NOTIFY` 由人工兜底。
        

### 3. T+1 离线对账功能 (T+1 Reconciliation) - _新增_

- **功能描述：** 每日凌晨自动核对前一日公司数据库记录与第三方平台真实资金流水。
    
- **核心步骤：**
    
    1. **账单抓取：** 自动调用微信/支付宝 API 下载前一日 `CSV` 资金账单。
        
    2. **数据清洗：** 将第三方账单转化为标准化的对账模型。
        
    3. **双向比对：**
        
        - **长款处理：** 外部有账但内部未标记支付。系统自动向业务方补发 MQ 通知或人工介入。
            
        - **短款告警：** 内部标记支付但外部无记录。**最高等级安全告警**，防范伪造回调攻击。
            
        - **差额核对：** 金额不匹配告警。
            
    4. **报告生成：** 生成每日对账汇总，作为财务入账的唯一依据。
        

### 4. 统一退款与查询 (Refund & Query)

- **功能描述：** 跨语言调用统一的退款接口，支持部分退款与原路退回。
    

## 五、 “极致标准” 非功能需求 (The Extreme Standards)

### 1. 性能 (Performance)

- **语言：** 必须使用 Go 编写，充分利用 Goroutine 处理高并发 Webhook。
    
- **延迟：** 网关内部逻辑处理延迟（不含网络 I/O）目标 **< 5ms**。
    
- **并发：** 单机支持 **10k+ QPS** 的回调压力，确保大促期间消息不积压。
    

### 2. 安全 (Security)

- **凭证管理：** 严禁数据库明文存储 API Key。采用 **AES-256-GCM** 加密，密钥由环境变量（K8s Secret 或环境变量）动态注入。
    
- **调用权限：** 下单与退款接口仅限内网 VPC 相互调用，外部 Webhook 接口强制进行 IP 白名单过滤或摘要算法校验。
    

### 3. 可靠性 (Reliability)

- **HTTP 回调可靠性：** 
    - 强制 3 秒超时，避免业务系统慢响应占用 GoPay 资源
    - 最多重试 5 次（指数退避：1s, 2s, 4s, 8s, 16s）
    - 失败后标记为 `FAILED_NOTIFY`，触发告警并由定时脚本或人工兜底
    
- **幂等保障：** 数据库 `(app_id, out_trade_no)` 唯一索引，结合 PostgreSQL 行锁（`SELECT ... FOR UPDATE`）处理并发 Webhook 通知。

- **业务侧幂等：** 业务系统必须做好幂等处理，因为网络抖动可能导致重复通知。
    

## 六、 数据库设计 (极简版)

- **Table: `apps`** (id, name, secret_key, configs_json, callback_url)
    
- **Table: `orders`** (id, app_id, out_trade_no, platform_trade_no, amount_cents, status, channel, notify_status, retry_count, created_at, paid_at)
    - `notify_status`: PENDING / NOTIFIED / FAILED_NOTIFY
    - `retry_count`: 重试次数，最大 5
    
- **Table: `recon_logs`** (id, date, status, diff_count, report_path)
    

## 七、 演进路线（分阶段实施）

### 阶段 1：MVP - 微信支付 + 单业务（2 周全职 / 4 周兼职）

**目标：** 跑通核心链路，服务第一个业务线

**交付物：**
- Go 核心框架 + PostgreSQL
- 微信支付 V3 Native 扫码支付（只做这一个渠道）
- Webhook 验签 + PostgreSQL 行锁并发控制
- 异步 HTTP 回调（带超时和重试）
- 接入第一个业务系统，完成真实交易测试

**技术选型：**
- 使用官方 SDK：`github.com/wechatpay-apiv3/wechatpay-go`
- 不要从零写签名验签逻辑

**可以省略：**
- ❌ 支付宝、银联、Stripe（后面再加）
- ❌ 自动化对账脚本（手工对账，每月一次）
- ❌ 管理后台（直接操作数据库）
- ❌ Prometheus 监控（用日志 + 钉钉/飞书告警）

### 阶段 2：多渠道 + 多业务（1 周）

**目标：** 验证架构的可扩展性

**交付物：**
- 支付宝接入（有了微信经验会快很多）
- 接入第二个业务系统
- 验证多 `app_id` 架构和回调隔离

### 阶段 3：开源准备（1-2 周）

**目标：** 按照内部开源标准重构，准备对外发布

**交付物：**
- 代码重构：遵循 golang-standards/project-layout
- 完善文档：README、API 文档、部署指南
- Docker Compose 一键部署
- 示例项目（Node.js/Go/Rust 接入示例）

### 阶段 4：生产加固（按需迭代）

**目标：** 提升稳定性和可观测性

**交付物：**
- 自动化 T+1 对账脚本
- 失败订单的人工处理工作台
- Prometheus + Grafana 监控
- 更多支付渠道（银联、Stripe）

**总计时间：**
- 核心可用（阶段 1）：2 周
- 生产就绪（阶段 1+2）：3 周
- 开源发布（阶段 1+2+3）：4-5 周
