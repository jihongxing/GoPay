# GoPay 年度维护指南

## 维护时间预估

### 第 1 年（开发年）
- 开发 GoPay：2 周
- 接入第一个业务：3 天
- 接入第二个业务：1 天
- Bug 修复和优化：1 周
- **总计：4 周**

### 第 2 年起（维护年）
- 官方 SDK 升级：2 次 × 2 小时 = 4 小时
- 证书更新提醒处理：1 次 × 1 小时 = 1 小时
- Bug 修复（如果有）：2 次 × 4 小时 = 8 小时
- 新业务接入：2 个 × 2 小时 = 4 小时
- 监控告警处理：10 次 × 0.5 小时 = 5 小时
- **总计：22 小时 ≈ 3 天/年**

### 极端情况（API 大版本升级）
- 微信支付 V3 → V4（假设发生）：1 周
- 但这种情况 5-10 年才发生一次
- 且官方 SDK 会提供迁移指南

## 突发事件处理

### 场景 1：微信支付 API 突然失败

**症状：**
- 下单接口返回错误
- 监控告警：下单成功率骤降

**排查步骤：**
1. 查看 GoPay 日志，找到具体错误信息
2. 检查微信商户平台是否有公告
3. 检查证书是否过期
4. 检查网络连接是否正常

**解决方案：**
- 如果是证书过期：立即更新证书（< 1 小时）
- 如果是 API 变更：升级官方 SDK（1-2 小时）
- 如果是微信平台故障：等待恢复，向业务方说明情况

### 场景 2：业务系统收不到回调通知

**症状：**
- 订单状态已更新为 PAID
- 但业务系统没有收到通知
- 监控告警：通知失败数量增加

**排查步骤：**
1. 查看 GoPay 日志，确认是否发送了通知
2. 查看业务系统日志，确认是否收到请求
3. 检查网络连接（GoPay → 业务系统）
4. 检查业务系统的回调接口是否正常

**解决方案：**
- 如果是网络问题：修复网络配置
- 如果是业务系统宕机：等待恢复，使用手动重试接口补发通知
- 如果是业务系统处理超时：优化业务系统逻辑，或增加超时时间

### 场景 3：数据库连接池耗尽

**症状：**
- GoPay 响应变慢
- 日志中出现 "too many connections" 错误

**排查步骤：**
1. 查看数据库连接数：`SELECT count(*) FROM pg_stat_activity;`
2. 查看是否有慢查询：`SELECT * FROM pg_stat_activity WHERE state = 'active' AND query_start < now() - interval '5 seconds';`
3. 查看是否有死锁：`SELECT * FROM pg_locks WHERE NOT granted;`

**解决方案：**
- 如果有慢查询：优化查询，添加索引
- 如果有死锁：重启 GoPay（事务会自动回滚）
- 如果连接池配置不合理：调整 `MaxOpenConns` 和 `MaxIdleConns`

## 官方 SDK 升级流程

### 微信支付 SDK 升级

**步骤：**
1. 查看 Release Notes：https://github.com/wechatpay-apiv3/wechatpay-go/releases
2. 确认是否有 Breaking Changes
3. 升级依赖：
   ```bash
   go get -u github.com/wechatpay-apiv3/wechatpay-go
   go mod tidy
   ```
4. 运行单元测试：
   ```bash
   go test ./pkg/channel/wechat/...
   ```
5. 在沙箱环境测试完整流程
6. 部署到生产环境
7. 观察 1-2 小时，确认无异常

**如果有 Breaking Changes：**
- 阅读迁移指南
- 修改 `pkg/channel/wechat/native.go` 中的调用代码
- 重新测试

**预计时间：**
- 无 Breaking Changes：1-2 小时
- 有 Breaking Changes：4-8 小时

### 支付宝 SDK 升级

**步骤：**
1. 查看 Release Notes：https://github.com/smartwalle/alipay/releases
2. 升级依赖：
   ```bash
   go get -u github.com/smartwalle/alipay/v3
   go mod tidy
   ```
3. 测试流程同上

## 证书更新流程

### 微信平台证书（自动更新）

**官方 SDK 会自动下载和更新平台证书，无需手动操作。**

**验证：**
```go
// 启动时会看到日志
// [INFO] 微信平台证书已自动更新
```

### 微信商户证书（每 5 年更新一次）

**提前 30 天会收到告警：**
```
[WARN] 商户证书即将过期，还有 30 天
```

**更新步骤：**
1. 登录微信商户平台
2. 进入"账户中心" → "API安全" → "申请证书"
3. 下载新证书（apiclient_cert.pem, apiclient_key.pem）
4. 加密存储到数据库：
   ```bash
   # 使用 GoPay 提供的 CLI 工具
   ./gopay-cli encrypt-config --app-id=xxx --cert-path=./apiclient_cert.pem
   ```
5. 重启 GoPay
6. 测试支付流程

**预计时间：** < 1 小时

## 降低维护成本的最佳实践

### ✅ 应该做的

1. **使用官方 SDK**
   - 不要自己写签名验签
   - 及时升级 SDK

2. **保持代码质量**
   - 核心逻辑有单元测试
   - 代码有清晰的注释
   - 遵循 Go 最佳实践

3. **完善的监控和告警**
   - 关键指标异常时立即告警
   - 日志完整，可以追踪每笔交易

4. **定期维护**
   - 每月检查一次
   - 每季度升级一次依赖
   - 每年做一次大检查

5. **文档更新**
   - 遇到问题时记录解决方案
   - 更新运维文档

### ❌ 不应该做的

1. **不要自己实现签名算法**
   - 这是维护成本的最大来源

2. **不要忽略官方公告**
   - 微信/支付宝的重要变更都会提前通知

3. **不要接入太多支付渠道**
   - 每多一个渠道 = 多一份维护成本
   - 够用就行

4. **不要忽略安全更新**
   - 官方 SDK 的安全补丁必须及时升级

5. **不要硬编码配置**
   - 所有配置都应该从环境变量或数据库读取

## 常见问题

### Q1：如果我长期不维护会怎样？

**短期（1-2 年）：** 可能没问题，但风险逐渐增加
- 官方 SDK 有安全漏洞
- 证书过期导致支付失败
- API 变更导致功能异常

**长期（3-5 年）：** 几乎肯定会出问题
- 微信/支付宝 API 大版本升级
- 依赖的库不再维护
- 安全漏洞被利用

**建议：** 至少每季度检查一次，每年升级一次依赖。

### Q2：我可以外包维护吗？

**可以，但不推荐。**

**原因：**
- 支付是核心基础设施，外包风险高
- 外包成本可能比 Ping++ 还贵
- 沟通成本高，响应速度慢

**如果真的要外包：**
- 选择可信赖的技术合作伙伴
- 签订明确的 SLA 协议
- 保留代码的完全控制权

### Q3：如果我不想维护了怎么办？

**迁移到 Ping++/BeeCloud：**
1. 保持 GoPay 继续运行
2. 新业务接入 Ping++
3. 老业务逐步迁移
4. 迁移完成后下线 GoPay

**预计时间：** 1-2 周

**成本：** 从此开始支付 0.2-0.5% 的额外费率

## 总结

**手搓 GoPay 的维护成本是可控的：**
- 正常情况：3 天/年
- 极端情况：1 周（5-10 年一次）

**关键是：**
1. 使用官方 SDK（最重要）
2. 保持代码质量
3. 完善的监控和告警
4. 定期维护

**如果做到这些，维护成本远低于 Ping++ 的额外费率。**
